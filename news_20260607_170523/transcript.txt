Anthropic 在五月初發了一份 AI 新創 Playbook
裡面有個詞叫做 ten-person unicorn
十個人的獨角獸公司
這句話聽起來很矽谷
但它背後講的其實是
AI 生產力爆發後的一種新公司形態
以前創業的過程大概是這樣
先用最小可行產品驗證市場
然後募資，然後擴編，然後做產品
然後再募資，再招更多人，再做更多功能
每往前走一步，公司就要變大
管理就要變重，成本就要變高
但 Anthropic 認為這條路徑正在被 AI 改寫
現在一個創業者可以用 Claude 做市場研究
用 Claude Code 做產品原型
用 Claude Cowork 處理營運流程
也就是說，以前那些需要顧問，運營， PM
或者是數據分析師的工作
現在可以被壓縮到一個很小的團隊裡面
甚至被壓縮到一個人身上
但聽到這邊，你不要誤會了
AI 不是讓創業變簡單
而是讓執行還有試錯的成本都變得更低
好的想法可以更快的被驗證
但錯的想法也可以很快就被包裝得有模有樣
以前工程成本很高
所以它某種程度上是創業者的煞車
你真的要花錢，花時間，找人
才有辦法把一個想法做出來
今天做出錯誤的產品
失控的功能變得更加容易
所以要如何與 AI 協作並產出真正安全
可用的產品這件事情就變得更加重要

如何在現有的工作中融入 AI
傳統創業幾乎在進入每個新的階段都要擴編
都需要更多的人
你要做研究，所以要找懂市場的人
你要做產品，所以要找工程師
你要做營運，所以要找 ops
你要做增長，所以要找行銷，找 BD
公司成長往往等於 headcount 成長
因為在過去，工作就是靠人堆出來的
但一間 AI-native startup 的狀況就不一樣了
當然並不是說你懂 AI 就永遠不用招人
但在很多階段
你可以先用 AI 把工作流跑起來
等到某個流程真的被驗證
真的有穩定需求，真的值得專人負責
再去招人
以前是先招人，再加入流程
現在可以先用 AI 跑流程
再決定哪些流程值得變成職位
這就是小團隊大公司的本質
不是十個人突然都變成天才
而是每個人背後都有一套 AI 系統在放大他的產出
但這件事情的陷阱在於
當 AI 放大了團隊的執行力
你的判斷力也會變得尤其重要
因為沒有判斷力的人
錯誤也會用更快的速度被放大
你本來有一個沒被驗證的想法
以前頂多寫在 Notion 裡，跟朋友講一講
現在你可以叫 Claude 幫你做出 demo
幫你寫 pitch deck
幫你搞 landing page
甚至連銷售信都可以一起生出來
做出來的東西或許看起來有模有樣
但它可能只是被 AI 包裝過的錯誤假設
那要怎麼用 AI 才不會掉進這個坑呢

我們進 AI 時代下的創業階段
這份 Playbook 把創業分成
Idea ， MVP ， Launch ，還有 Scale 這四個階段
我們先從第一個 Idea stage 開始
Idea stage 這個階段最重要的
不是做產品，是做驗證
在這個階段， AI 最大的用途是市場研究
競品分析，反向挑戰你的假設
避免做出一個沒人要的東西
Playbook 裡叫創業者
一開始要回答五個關鍵問題
第一個問題是，這個痛點是不是真的存在
第二個是誰有這個問題
第三是這個問題發生得夠頻繁嗎
第四是現在大家怎麼解決
最後是你的解法
到底有沒有打中真正的問題
這些問題聽起來很基本，但最容易被跳過
因為現在做一個東西太容易了
裡面引用了一個經典數據
CB Insights 的資料裡
42% 的 startup 失敗
是因為 built something nobody wanted
也就是做了一個沒人要的東西
AI 不會自動解決這個問題
AI 只會讓你更快做出一個沒人要的東西
所以 Idea stage 第一個 use case
不是叫 Claude 幫你把 idea 寫漂亮
而是叫 Claude 幫你把 idea 變成可以驗證的假設
如果今天的任務是
把一句模糊的創業想法
變成一個可以驗證的問題假設
你給 Claude 的輸入
不應該只是我想做一個 AI 工具
幫公司提升效率這種廢話
你要給它的是，你觀察到的痛點
你想服務的人，他們現在的工作流程
你懷疑哪個地方有摩擦
然後請 Claude 幫你拆成剛剛提到的五個問題
最後你要判斷這些問題是不是可以一一回答
如果你講不出受眾是誰
講不出發生場景
講不出市面上有的替代方案
那它就不是一個創業 idea
它只是你的一時興起
第二個 use case 是反向驗證
大部分人用 AI 做市場研究
會犯一個很自然的錯誤
就是跟 AI 說
請幫我分析這個市場為什麼有機會
你這樣問，它真的會給你一份很完整
很有說服力
看起來很像 McKinsey 報告的分析
因為它在幫你找支持證據
就像你跟他說請證明猴子喜歡喝寶礦力一樣
他會想辦法睜眼說瞎話，儘管沒有邏輯
創業早期你最不缺的就是支持證據
因為 founder 本來就會愛上自己的 idea
你缺的是有人用理性的方式告訴你
你以為的待開發市場為什麼根本不成立
所以更好的任務是
請 Claude 扮演反方
找出這個 idea 最可能失敗的原因
輸入資料可以是你的問題假設，解法草案
目標市場，還有你知道的競品
然後請它輸出幾類東西
第一，客戶為什麼可能根本沒有這個痛點
第二，競品或替代方案為什麼已經夠好
市場為什麼可能不需要你
第三，這個市場為什麼可能太小
沒有你的容身之地
第四，這個解法即使做出來
為什麼也很難賣
founder 要判斷的不是 Claude 說得對不對
founder 要判斷的是
哪些反對意見是真的會讓你改變方向
哪些只是你需要承擔的風險
這才是 AI 在 Idea stage 最有價值的地方
第三個 use case
是 customer discovery 訪談設計
很多創業者訪談客戶，其實是在問廢話
例如，如果我做一個可以幫你節省時間的工具
你會不會用
這種問題沒有意義
大部分人都會說會啊
因為聽起來沒成本，也不用現在掏錢
真正有用的問題，是問過去，不是問未來
不要問你會不會用
要問你上一次遇到這個問題是什麼時候
你當時是怎麼處理的
花了多久，有沒有花錢解決
如果沒有，為什麼
這裡 Claude 可以做的事
是幫你設計訪談問題
然後幫你檢查哪些問題太具備誘導性
所以答案會有所偏頗
以及又有哪些問題太過抽象
你給它目標受眾的 persona
你想驗證的關鍵假設
它輸出訪談問題清單，追問問題
以及每個問題背後在驗證什麼
這場訪談是在蒐集真實行為證據
還是在找人認同自己
這三個 use case 都指向同一件事
Idea stage 的 Claude ，不是你的啦啦隊
它應該是你的研究員，也是你的反方辯論手
如果一個 idea 經不起這一關
改變方向或許是最正確的選擇
接下來進到第二階段， MVP stage
MVP 的全文是 Minimum Viable Product
中文叫做最小可行性產品
這一段是讓很多人最興奮的地方
因為 AI 的出現
讓打造出一個可以快速驗證市場的 MVP
變得超級無敵快
但我認為這一段也最容易出事
因為 AI coding 最大的問題
不是它寫不出來，而是它太願意寫
以前 scope creep 會被工程成本擋住
你想加功能，工程師會皺眉
PM 會排優先級，老闆會看 budget
現在沒有這些摩擦
你的每個突發奇想都可以很快變成產品的一部分
你今天覺得 onboarding 要多一步
明天覺得 dashboard 要多一個圖
後天覺得應該加一個 admin panel
AI 都可以配合
聽起來很棒，但實際上這就是無意義的需求蔓延
AI 最擅長的是 greenfield project
也就是從零開始做一個 demo
這時候它很快，很猛，很有成就感
但真正麻煩的是 brownfield project
也就是產品已經有人用
資料已經進來
架構已經開始累積技術債之後
你還要持續迭代
這時候如果前面沒有 scope ， spec ， context
AI 不是幫你加速，而是幫你把混亂放大
Playbook 裡面提到一個重點
MVP stage 不是完整產品建設
它還是 evidence-gathering
用白話講，還是在蒐集證據
所以你要證明的是
一群具體使用者真的覺得這個東西有價值
會回來繼續使用，會付費使用
甚至會推薦給身旁的朋友
所以第一個 MVP use case
是 MVP scope doc
你可以請 Claude Code 開始寫之前
先定義這個 MVP 做什麼，不做什麼
你可以請 Claude 幫你產出一份 scope doc
裡面包含核心功能
未來疊加新功能的條件有哪些
比方說要達到多少月活用戶
以及最重要的排除功能
也就是 MVP 版本我們不做哪些功能
你一定要寫這塊
因為 AI coding 的年代
不做會變得比做更重要
founder 要有判斷力
去決定這個功能是不是能夠驗證產品價值的核心
還是只是手癢想加
第二個 use case
是做一個給 Claude code 的 CLAUDE markdown
Playbook 裡面講得很清楚
AI-native startup 的 codebase
是你會一個 session 接一個 session
跟 AI 一起協作的東西
如果你沒有 specs ，沒有 context files
每次開新 session
AI 都會重新推導一次這個系統應該長什麼樣
久了以後
因為整個系統沒有一致的心智模型
你的 codebase 會開始漂移
而 CLAUDE markdown 的作用
就是讓 Claude Code 每次進來
都知道這個專案的基本規則
就像我們的校規，像員工手冊
是一份給未來 AI session 看的工作記憶
是讓 AI 在結構上保持一致性
而不是每次都在用局部最佳解把產品拼起來
在迭代產品時代碼崩潰，全部重來
是上線前的 security review
AI 寫出來的程式碼，能跑不等於安全
功能對不對你一眼就看得出來
會動就是會動，但安全漏洞不會報錯
不會當機
它就靜靜躺在那裡
等到有人利用它你才知道
而越不會寫 code 的 founder ，這個坑越深
因為你根本不知道自己漏在哪
所以任務很簡單
在任何真實使用者碰到產品之前
先讓 Claude 幫你掃一輪
重點看 authentication 跟 session 處理
API 會不會把洩漏機密資料
或者有沒有 injection 風險
但要記住
Claude 幫你做的是第一輪檢查
不是你的唯一防線
真的碰到使用者資料，碰到金流的部分
該上專業工具，該找人看，還是不能省
最後出現在資安事故新聞裡的
往往就是把 AI 當成唯一防線的那種人
總的來說，第二階段 MVP stage 真正的核心是
用 Claude Code 快速做產品
但用 scope ， context ， security 把它管住
速度本身不是優勢，可控的速度才是
如果你正處於創業階段
我把完整文章放在 Patreon
文章中有搭配一套提示詞工具包
可以幫你檢查自己的 MVP ， AI workflow 等
有需要的朋友可以在影片下方資訊欄找到連結
好，那我們繼續
再往下，第三階段是 Launch stage
Launch 不只是產品上線
發一篇文章或開始打廣告
在 Anthropic 這份 Playbook 裡
它更像是公司開始從 founder 手工操作
變成一套可以重複運轉的系統
這個階段的問題
不再只是產品能不能用
而是 growth ， feedback ， metrics
security ， ops
這一切的一切能不能不要每件事都卡在 founder 身上
MVP 階段， founder 在每個 loop 裡面是優勢
因為你要貼近使用者
要快速感受問題
要親自判斷哪些 feedback 是真的
但到了 Launch 階段
如果所有事情都還要你親自記得
親自整理，親自判斷，親自追
你就會變成效率上的瓶頸
AI 真正有價值的地方
不是偶爾幫你做一件事
偶爾叫 AI 幫你寫一封信，整理一篇文章
有用，但價值有限
更重要的是
你能不能把重複出現的工作流
變成一套固定會運轉的系統
如果不知道怎麼把附近的工作流程拆解成
Agent 可以執行的 Workflow
我上週有出一部很詳細的教學影片
各位可以去看看
比方說資訊進來以後，誰要整理
有 to-do list 以後，誰要負責追蹤
回饋累積以後，誰來歸納並落地
這些事情如果每次都靠人想起來，就不是系統
Launch stage 的 AI
不應該只是聊天機器人
而應該開始變成營運系統的一部分
一樣給幾個 use case
一個是 weekly metrics brief
Launch 之後，會有很多數據出現
activation ， retention ， usage
bug count ， conversion funnel
很多 founder 會被 launch spike 騙到
剛上線的時候，朋友支持，社群曝光
早期來自好奇心的流量
都會讓數字看起來不錯
但問題是，六週後，十二週後
這些人還在不在
有沒有付費，有沒有持續使用產品
所以你要讓 AI 做的任務是
讓它固定產出每週 metrics brief
輸入資料是各種產品和營運指標
AI 產出本週變化，異常訊號
可能原因，下週應該追的問題
讓你可以直觀的判斷
這些數字是在反映真實 product-market fit
還是在反映短期熱鬧
Playbook 裡也提到 Sean Ellis test 可供參考
這是一個用來評估產品
是否達到產品市場契合度的調查方法
運行方式是問活躍使用者
如果你不能再用這個產品
你會有多失望
如果超過 40% 回答 very disappointed
這是一個有意義的 PMF 指標
第二個 use case 是 support 和 bug triage SOP
產品開始有使用者以後，問題會一直來
有些是 bug
有些是 onboarding 不清楚
有些則是產品設計本身有問題
如果每一個問題都直接丟給 founder 或工程師
這家公司很快就會被小事淹沒
所以任務是，用 Claude 產出一套 triage SOP
幫你建立分類規則，優先級
標準回覆，升級條件
讓你可以看懂
哪些問題可以流程化
哪些問題不能只靠流程處理
因為它們代表產品本身需要改
這就是 Launch stage 的重點
AI 不只是幫你做更多事
AI 是幫你把公司裡反覆出現的工作
變成流程，過濾噪音，協助你快速判斷
最後一個階段是 Scale stage
它決定一間 AI-native startup
最後到底有沒有護城河
還是只能曇花一現
到這個階段
founder 的角色會從做產品的人
慢慢變成對外的經營者
但你真正的優勢
還是那條 lean ，用 AI 撐起來的結構
Playbook 在這裡點出三個很關鍵的東西
我一個一個白話講
第一個是 accumulated depth ，累積出來的深度
意思是你的護城河來自你對這個領域的理解
你的產品跟客戶其他工具整合的深度
還有你手上那些別人沒有的資料和流程
第二個是 proprietary system data
你獨有的系統資料
你的產品被用得越久
你就越懂這群人怎麼做事
而這些細節
不是競爭對手看你的 landing page 就能抄走的
第三個是 workflow lock-in ，工作流綁定
當客戶把一堆自動化，團隊習慣
標準輸出全都建在你的產品上面
要換掉你就不再是按一個取消訂閱這麼簡單的事
而是一個 operational project
這三個東西從哪來，我舉兩個例子
第一個是 user behavior data flywheel
使用者行為的數據飛輪
你要從使用者行為裡找出
可以持續改善產品的訊號
例如哪些 output 被改掉
哪些流程被重複使用
哪些功能根本沒人碰
持續追蹤
把這些訊號轉成優化的 idea
模型改善， workflow 改善
產品越用越準，越準就越多人用
飛輪就轉起來了
第二個例子是
一個產品最容易被替換的狀態
就是它只有一個功能
今天你做摘要，明天別人也做摘要
你沒有技術壁壘
使用者換掉你也沒什麼成本
但如果你的產品已經接到他的資料來源
進入他的團隊流程
影響他的標準輸出
那他的離開成本就會高到不像話
所以 Scale 階段一個很值得做的 use case
是讓 Claude 幫你做一張 workflow 盤點
你把使用頻率，團隊協作流程
客戶依賴的模板和輸出格式這些資料餵給它
請它幫你整理出
不同客戶群各自把你的產品用得多深
哪些是最難被替換的節點
以及未來還可以從哪裡加深整合
最後， Anthropic 這份 AI 新創 Playbook
表面上是在講 Claude 可以怎麼幫創業者
但我認為最關鍵的事情在於
未來新創公司的差距
不是誰有沒有用 AI
這件事很快就不會是差異了
真正的差距
是誰把 AI 系統化得更好
誰能把 AI 放進每個階段的工作流程
把 AI 放進營運流程
讓 feedback ， metrics ， support
不再全部卡在 founder 身上
把 AI 放進 scale 流程
把資料， workflow 沉澱成產品護城河
這些才是真正的差距
AI 會放大創業者的判斷力
也會放大創業者的愚蠢
一個懂得驗證，懂得設定邊界
懂得建立 workflow 的 founder
會用 AI 加速學習和執行
能夠用最快的速度自我迭代
讓團隊適應這個市場然後存活下去
一個沒有判斷力的 founder
只會用 AI 更快做出錯的東西
所以我看這份 Playbook
最重要的 takeaway 不是 Claude 有多強
而是創業者要開始把公司拆成一條條流程
哪些需要研究，哪些需要建構
哪些需要營運
哪些需要沉澱成護城河
然後再問 AI 在這條流程裡
應該扮演什麼角色
這才是 AI-native startup
不是用了 AI 工具的公司
是把 AI 變成工作系統的公司
