# 提示詞越寫越長反而害了你！Anthropic工程師：Claude Code團隊砍掉8成系統提示，模型才真正發揮

## 📖 新聞導語

2026 年 7 月 1 日，Anthropic 技術團隊成員、Claude Code 團隊工程師 Thariq Shihipar 在 AI Engineer World's Fair 發表《Field Guide to Fable》演講。他以「Fable is back」開場，分享與新一代模型協作的經驗，並拋出核心觀點：模型表現好不好，往往不只看模型本身，改善工具環境與提示更能釋放其潛藏實力。

## 💡 核心內容

### 系統提示詞：從「禁令」到「脈絡」
Shihipar 回顧了系統提示的演變：Claude 3.5 Sonnet 時代，最佳做法是「提示寫短、工具給少、多塞範例」；一段時間後流行反過來「提示寫長、工具給多、範例塞好塞滿」；但到了最新一代，又倒回去——模型反而想要更短的提示，因為範例會綁住它的想像力。

Claude Code 團隊近期把系統提示砍掉了大約八成，原則從「規定它不能做什麼」改成「給它足夠的背景」，少下禁令、多給脈絡。

### 模型與工具共同成長
他舉例說明：一般聊天模型問「名字以『aw』結尾的寶可夢有哪些」，未必能答對；但交給 Claude Code，它能自己跑程式、抓出完整清單再篩選，找出 Croconaw（藍鋸鱷）和 Drednaw（暴噬龜）。這說明差別不一定只來自模型聰不聰明，也可能來自它能不能動手用工具。

另一個例子是「讓模型反過來問你問題」：在 Opus 4 上只能勉強用；到了 Opus 4.5，能一口氣完成一輪像樣的訪談；再到 Opus 4.8 和 Fable 5，它甚至能直接生出一份把問題嵌在裡面的 HTML 報告。

### 瓶頸轉移到人類身上
當模型被「解開束縛」後，瓶頸並沒有消失，而是轉移到人身上。他引用老話「地圖不等於實地」：你腦中的計畫、提示和規格是地圖；真正的程式庫、現實限制才是實地。Claude Fable 5 是最早讓他覺得「必須先把未知想清楚」的模型之一，因為它會一口氣跑很大一片範圍，撞上很多你地圖上沒畫的決策點。

他把「你知道什麼」分成四種：
1. 你知道的
2. 你不知道的
3. 你其實知道但沒特地寫下來（看到就認得）
4. 你完全沒想到的

## 🔮 業界展望

Shihipar 留給全場 AI 工程師最重要的一句話：「**東西變好做了，但做出價值還是難（building is easier, but generating value is still hard）**。」他提醒別太沉迷於把流程和環境弄得漂漂亮亮，那只是「變好做」的部分；真正難的，是找到那件值得做的事，而那通常要試很多次才會浮現。對於 AI 工程師而言，如何找出自己沒想到的問題，已成為新的核心挑戰。
