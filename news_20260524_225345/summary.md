# 我的 AI agent 連續跑了 27 個小時，/goal 功能怎麼用？ - Gary Chen

## 📖 新聞導語

**Gary Chen** 在上週進行了一個驚人的實驗：讓 AI agent 連續跑了 **27 個小時**，過程中沒有人為介入。這支影片深入解析了 Claude Code、OpenAI Codex、Hermes Agent 同步推出的 `/goal` 功能，以及如何通過「評審 + 執行者」的雙 agents 架構，讓 AI 真正做到長時間自主運行。

## 💡 核心內容

### 🤔 問題：為何 AI 總是半途而廢？

你一定遇過這種情況：叫 AI 做一件事，它做到一半就停下來問「我可以繼續嗎？」或者更糟——明明沒做完，卻跟你說做完了，寫了漂亮的總結把球丟回給你。

**Anthropic 2025 年底的研究** 揭露了根本原因：LLM 有所謂的「**下班心態**」——當它感覺到 context 快滿了，就會開始慌，自動想要 wrap up 交差了事。

### ⚙️ 解決方案：`/goal` 功能

三間公司（Claude Code、OpenAI Codex、Hermes Agent）在同一時間推出了同名功能 `/goal`：

**運行原理：雙 agents 協作**
- **執行者（Actor）**：負責執行指令、產出內容
- **評審（Reviewer）**：在每一輪結束後檢查「目標完成了嗎？」——如果答案是否定的，就點出問題並叫執行者繼續

就像把胡蘿蔔吊在驢子前面——**驢只要還沒吃到，就不會停下來**。

### 📝 如何寫好一個 Goal Prompt？

爛的例子：
> 把這個專案改得好一點

好的例子（包含五個關鍵元素）：
> 把網站結帳頁面的反應速度降到 0.2 秒以內，用速度測試工具驗證，過程中其他功能要保持完好，只能改結帳區塊相關程式碼，每改一次記錄改了什麼、下一步最值得試的方向，如果工具跑不起來或所有方法都試過了就停下來告訴我你試過什麼

**五個關鍵元素**：
1. **Outcome** — 任務完成的定義（反應速度 < 0.2 秒）
2. **Verification** — 如何驗證（速度測試工具）
3. **Constraints** — 限制條件（只能改結帳區塊）
4. **Iteration policy** — 每次嘗試之間做什麼（記錄改動、結果、下一步）
5. **Error handling** — 什麼情況要停下來回報

### 🎨 Anthropic 的研究：如何評估「品質」

對於沒有單元測試的白領工作（如網頁設計、寫作），Anthropic 把「做得漂亮」這個模糊概念拆成四個維度：

1. **設計品質**：顏色、字體、排版是否營造整體氛圍
2. **原創性**：有沒有刻意設計選擇，還是用了一堆預設模板
3. **技術執行**：字體階層、間距、對比是否整齊一致
4. **可用性**：使用者能不能直覺完成目標

更有趣的是——他們發現 Claude 在設計品質和原創性上特別容易產出平庸作品，所以故意把評審權重往這兩個弱項偏。**每個模型有自己的傾向，評分標準可以校正模型的預設行為**。

第 10 輪時，Claude 竟然把整個網站重新想像成一個 3D 空間體驗——連研究人員都說這是不可能從單次 prompt 產出的創意躍遷。

### 📋 六步驟 SOP：把品味轉化為 AI 可以遵守的標準

1. **先讓 AI 做一輪**：不要急著寫標準，先看 AI 的 baseline 能力
2. **親自看產出，感受皺眉點**：把「皺眉的具體原因」寫下來，這是 rubric 雛形
3. **把皺眉理由分類成維度**：如「邏輯鬆散」「沒有人味」「開頭沒 hook」
4. **每個維度寫成具體案例**：絕對不要用破折號、絕對不要用「不是 A 而是 B」句型
5. **用多樣化案例取代單一描述**：Anthropic 說「博物館質感」會讓產出都變成博物館風，後來改列出 11 種美學風格才解決
6. **餵給評審 agent 跑跑看**：如果評審的判斷跟你親眼看的結果不一致，就回去改 rubric
