# LLM 每次處理都會腐蝕你的文件：DELEGATE-52 研究揭示delegation workflow的致命缺陷

## 📖 新聞導語

**來源**：Hacker News / arXiv　**發布日期**：2026-05-10

一篇名為 DELEGATE-52 的學術論文首次系統性地研究了一個長期困擾 LLM 用戶的問題：每次通過 LLM 處理文檔都會導致內容降解。研究人員模擬了 52 個專業領域的長delegation workflow，結果發現即使是最強的前沿模型也會平均腐蝕 25% 的文檔內容。

## 💡 核心內容

### DELEGATE-52 研究

研究人員構建了一個delegation workflow模擬系統，研究 AI 系統在長 workflow 中的表現。測試涵蓋 19 個 LLM，包括 Gemini 3.1 Pro、Claude 4.6 Opus、GPT 5.4 等前沿模型。

**主要發現**：
- 即使是最強模型，在長 workflow 結束時平均腐蝕 25% 的文檔內容
- 其他模型失敗率更高
- 每次 LLM 處理都會導致訊息損失和扭曲

### 專家解讀

**「LLM 是最複雜的猜測機器」** — LLM 每次處理都像對圖片進行 JPEG 壓縮，每次都會損失一些品質，最終變得無法識別。

**核心比喻**：
- LLM 像「電話遊戲」—— 訊息每次傳遞都會有些改變
- 文件就像 JPEG 圖片—— 每次壓縮都會降解
- 意圖就像熵—— 每次 LLM 處理都會增加混亂

### Hacker News 社群反應

**支持研究觀點**：
- 「這是一個主要問題」——律師表示在處理法律文件時被告知要大量使用 AI，導致錯誤發生
- 「期望 30 秒內掌握是不合理的」——批評者忽略了 LLM 需要學習和練習的事實
- 「LLM 是均值回歸機器」——越偏離訓練數據的內容，越容易被拉向平均

**批評研究方法**：
- 研究使用的 agentic harness 太基礎（只有 read_file 和 write_file），相當於「每次都是 round-trip」
- 現代 coding agent（如 Claude Code）使用專門設計的工具來避免這個問題
- 專家認為結果更多反映的是 harness 設計問題而非模型本身問題

### 為何這很重要

1. **企業應用**：大量非技術用戶每天使用 LLM 處理文檔，不知道這個問題
2. **模型崩潰（Model Collapse）**：當 LLM 輸出被反覆餵回訓練數據時，會導致知識退化
3. **長期累積效應**：每次小錯誤都會累積，最終導致大規模偏差

### 實用建議

| 方法 | 說明 |
|------|------|
| **使用 surgical edit** | 只讓 LLM 做小範圍精確編輯，而非整個文檔來回處理 |
| **隔離事實與 LLMs** | 將確定性數據通過 LLM 處理前先驗證，LLM 只輔助寫作而非數據處理 |
| **建立版本控制** | 使用 git diff 追蹤 LLM 的變更，並手動審查 |
| **分開處理** | 讓 LLM 研究獨立的 markdown 文件，最後再組裝成文檔 |
